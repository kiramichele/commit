The editor

Writing and running code

The editor works like any basic code editor — you type, you run, you see what happens. A few things worth knowing:

1
Type your code in the editor

The editor knows you're writing Python. It highlights keywords, catches obvious syntax errors, and shows line numbers. You don't need to install anything — it runs in the browser.

2
Hit Run (or Ctrl + Enter)

Your code executes in a sandboxed Python environment. Output appears in the panel below the editor. Errors appear there too — in red, with the line number that caused them.

3
Read the output — including errors

An error message is not a failure. It's the computer telling you exactly what went wrong and where. Learning to read error messages is one of the most important skills you'll build this year. We'll practice it deliberately in Lesson 3.8.

4
Iterate until it works

Real programmers run their code dozens of times while writing it. Every run is a test. You're not expected to get it right on the first try — that's not how programming works for anyone.

The commit system

What a commit is — and why it matters

The word "commit" comes from version control software — the same tools professional developers use to track changes to their code over time. When you commit in this platform, you're saving a named checkpoint: a snapshot of your code at that moment, with a message describing what you did.

Over the course of a project, your commit history tells a story. It shows how you started, where you got stuck, how you figured it out, and what you changed. Your teacher can see this history — not to judge every step, but to understand your thinking process, not just your final answer.

Why this exists

On the AP Create PT, you'll be asked to explain your program's development process. Students who have been committing throughout the year have a built-in record to reference. Students who only save at the end have to reconstruct it from memory. Commit early. Commit often.

Commit messages

Writing a good commit message

A commit message is a note to your future self — and to your teacher. It should describe what you changed and why, not just acknowledge that something happened. Here's the difference in practice:

Not useful
done
fixed it
asdfgh

These tell you nothing about what changed. In six weeks, "fixed it" is useless — you won't remember what "it" was.

Actually useful
added input validation — now handles empty strings
fixed off-by-one error in the loop counter
first working version of the tip calculator

These describe a specific change. Anyone (including you, later) can read the history and understand what evolved.

You don't need to write a paragraph. One clear sentence is enough. The habit of writing descriptive commit messages is something professional developers take seriously — it's one of the first things a code reviewer looks at.

Submitting work

How submission works

Committing and submitting are two different things. You can commit as many times as you want while working — think of commits as drafts. When you're ready to officially hand in an exercise or assignment, you use the Submit button.

What your teacher sees

Your teacher's view

It helps to understand what's visible on the other side. Your teacher has a dashboard showing every student's current progress — which lessons are complete, which exercises are submitted, and the commit history for each one.

This isn't surveillance. It's how your teacher can tell when someone is stuck without them having to ask. If you've made five commits that all say "trying again" and your code still isn't working, your teacher can see that and check in — before you spend 45 minutes frustrated on something that has a simple fix.

Feedback

Where to find feedback on your work

When your teacher reviews your submission, feedback appears directly on the exercise page — inline with your code if it's a specific line, or as a general comment at the top. You'll get a notification when new feedback is available.

Feedback isn't just "good job" or a grade. It's notes on your approach — why something works, why something doesn't, and what to think about next. Take the time to read it before you move on.

One rule for this class

If you're stuck for more than 20 minutes and your commit history shows you've genuinely tried — leave a comment in your code describing what you've attempted, commit it, and message your teacher. That's not giving up. That's knowing when to ask for help, which is a skill most people take years to develop.

Quick reference

Keyboard shortcuts worth knowing

Ctrl+↵
Run your code

Executes whatever is in the editor. Same as clicking the Run button.

Ctrl+S
Save a draft

Saves your current code without committing. Use this constantly.

Ctrl+/
Comment / uncomment a line

Toggles a # at the start of the selected line. Useful for temporarily disabling code to test something.

Tab
Indent

Python uses indentation to define code blocks — it's not optional decoration. The editor will try to auto-indent, but knowing how to manually indent is essential.

Tip

Python is very sensitive to indentation. A single misplaced space or tab inside an if block or for loop will cause an IndentationError. If your code isn't running and you can't see why — check the indentation first.
