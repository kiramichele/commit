Student submission: weather mood tracker

This is a fictional student submission written to match real AP CSP Create PT requirements. The program is a weather-based mood tracker — the student describes it below in their own words.

Written response 3a — Program purpose and function Prompt 3a
i. Describe the overall purpose of the program.

My program helps people track how the weather affects their mood over time. A lot of people don't realize that weather has a real impact on how they feel, so my program lets them log their mood alongside the current weather conditions each day. Over time they can look back and see patterns — like noticing they always feel worse on cloudy days or better when it's warm.

ii. Describe what the program does to accomplish its purpose.

The program asks the user to enter today's weather condition (sunny, cloudy, rainy, or snowy) and their current mood on a scale of 1 to 10. It stores each entry in a list. When the user wants to see their data, the program calculates the average mood for each weather type and displays the results. The user can keep adding entries or view their averages at any time.

iii. Describe the input and output of the program.

Input: the user enters a weather condition (a string) and a mood score (a number from 1 to 10). Output: the program displays the average mood score for each weather type that has been logged, along with how many entries exist for each type.

Written response 3b — Data abstraction (code segment 1) Prompt 3b
i. The list being used to store data
mood_tracker.py — segment 1
# List of entries — each entry is [weather, mood_score] entries = [] weather = input("Enter weather (sunny/cloudy/rainy/snowy): ") mood = int(input("Enter mood score (1-10): ")) entries.append([weather.lower(), mood])
ii. Explain what the data in the list represents.

The list called entries stores all the mood log entries the user has made. Each item in the list is itself a list with two elements: the weather condition as a string and the mood score as a number. Without this list I would have to create a separate variable for every entry, which would mean I could only store a fixed number of entries and the code would get very repetitive. The list lets the program store any number of entries using the same structure.

Written response 3b — Managing complexity (code segment 2) Prompt 3b cont.
i. How the list manages complexity
mood_tracker.py — segment 2
def get_average_mood(entries, weather_type): matching = [e[1] for e in entries if e[0] == weather_type] if len(matching) == 0: return None return sum(matching) / len(matching) weather_types = ["sunny", "cloudy", "rainy", "snowy"] for w in weather_types: avg = get_average_mood(entries, w) if avg is not None: print(f"{w}: average mood {avg:.1f} ({len([e for e in entries if e[0]==w])} entries)")
ii. Explain how this list manages complexity

The list manages complexity because it allows the same loop and the same function to work no matter how many entries the user has logged. If the program used individual variables instead, it would need to check each one separately — the code would be hundreds of lines long if someone logged 50 entries. With the list, adding a new entry is just one line (entries.append(...)) and the existing loop automatically includes it in every calculation.

Written response 3c — Procedural abstraction Prompt 3c
i. The student-defined procedure
mood_tracker.py — segment 3
def get_average_mood(entries, weather_type): """Returns the average mood score for a given weather type, or None if no entries exist for that type.""" matching = [e[1] for e in entries if e[0] == weather_type] if len(matching) == 0: return None return sum(matching) / len(matching)
ii. Describe what the procedure does and how it contributes to the program

The get_average_mood function takes the full list of entries and a weather type as parameters. It filters the list to find only entries that match the given weather type, then calculates and returns the average mood score for those entries. If there are no matching entries, it returns None so the rest of the program knows to skip that weather type. This function is called four times in the main program — once for each weather type — which means the averaging logic only needs to be written once.

Written response 3c — Algorithm implementation Prompt 3c cont.
i. The algorithm including sequencing, selection, and iteration
mood_tracker.py — segment 4
def get_average_mood(entries, weather_type): matching = [e[1] for e in entries if e[0] == weather_type] if len(matching) == 0: return None return sum(matching) / len(matching)
ii. Explain the algorithm in enough detail that someone could reproduce it

The algorithm works as follows: first, it uses iteration (a list comprehension) to go through every entry in the entries list and collect the mood scores where the weather type matches the given parameter. This produces a new list called matching. Then, using selection, it checks whether matching is empty — if it is, the function returns None and stops. If matching is not empty, the function calculates the average by dividing the sum of all scores by the number of scores, and returns that value. The result is a single number representing the average mood for that weather type.

Written response 3d — Testing Prompt 3d
Describe two calls to your procedure with different arguments and different expected results

Call 1: I called get_average_mood(entries, "sunny") after adding three sunny entries with mood scores of 8, 9, and 7. I expected the function to return 8.0, and it did.

Call 2: I called get_average_mood(entries, "snowy") when no snowy entries had been added yet. I expected the function to return None so the display loop would skip it, and it did. This tested the edge case where a weather type has no data.
