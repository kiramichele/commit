Every program ever written — from a two-line Hello World to the software running a Mars rover — does the same fundamental thing. It takes something in, does something with it, and produces something out. That's it. Understanding this pattern is the foundation everything else in programming sits on.

This lesson isn't about syntax. You're not going to write any code here. What you're going to build is a mental model — a way of thinking about what programs are doing even before you see a single line of Python. Students who have this model learn to code faster, debug more effectively, and write better programs. Students who don't tend to memorize syntax without understanding what it means.

The right mental model

A program is a recipe for a computer

A recipe is a list of precise instructions that anyone can follow to produce a predictable result. It specifies ingredients (inputs), steps to combine them (process), and what you end up with (output). A program works the same way — except the "anyone" is a machine that follows instructions with perfect literalness and zero tolerance for ambiguity.

Fig. 1 A recipe and a program are structurally identical: both take inputs, specify a precise sequence of steps, and produce a predictable output. The difference is that a recipe tolerates improvisation. A program doesn't.

Here's a recipe mapped directly to CS concepts:

Recipe language
CS equivalent
"2 cups flour, 1 egg, 1 tsp salt"
Input

"Mix flour and salt. Beat egg separately. Combine."
Process
The steps — instructions that transform the inputs into something new
"One loaf of bread"
Output
The result — what the program produces and gives back to the user
The crucial difference

A recipe says "add salt to taste." A program cannot. Every instruction a computer follows must be completely unambiguous — there's no room for judgment calls, approximations, or "you'll know it when you see it." This is why programming is hard: not because computers are complicated, but because humans are used to communicating with gaps that other humans fill in automatically. Computers don't fill in gaps. They fail.

Notice something: the complexity of the program doesn't change the pattern. A two-line tip calculator and a billion-dollar search engine both take input, process it, and produce output. The process step is where all the interesting work happens — and that's where you'll spend most of your time as a programmer.

Under the hood

What actually happens when you run code

When you hit Run in the Commit editor, you're asking a computer to execute a sequence of instructions. Here's what actually happens — simplified, but accurate:

1
Your code is read line by line

Python reads your program from top to bottom, one line at a time. It doesn't read ahead — it executes each line before moving to the next. This is called sequential execution, and it's why the order of your code matters enormously.

2
Each instruction is translated to something the CPU understands

Your Python code isn't what the computer actually runs. Python translates your code into much simpler instructions that the processor can execute directly — things like "add these two numbers" or "store this value in memory location X." You never see this translation, but it's happening.

3
Variables are stored in memory

When you write name = "Kira", Python reserves a small piece of your computer's RAM and stores the value "Kira" there, labeled with the name name. Every time you use name in your program, Python goes and fetches that value from memory.

4
Output is sent to the screen

When your program calls print(), it sends text to the standard output — the output panel in your editor. This is the simplest form of output. More complex programs write to files, send data over a network, or draw on a screen.

5
The program ends — or it doesn't

A program stops when it runs out of instructions, or when it explicitly exits. Some programs (like operating systems or web servers) are designed to run continuously — they loop forever, waiting for new input. Others run once and stop. Both are valid — it depends on what the program needs to do.

Why we need languages

Why programming languages exist

Computers don't understand English. They understand binary — sequences of 1s and 0s that represent electrical signals. Writing software directly in binary would look like this:

01001000 01100101 01101100 01101100 01101111 00101100 00100000 01110111 01101111 01110010 01101100 01100100

That's "Hello, world" in binary. Nobody writes software this way — it would be impossibly slow and error-prone. Programming languages like Python exist as a middle layer: human-readable enough that you can think in them, but close enough to machine instructions that they can be automatically translated.

Python sits at the "very readable" end of the spectrum. It was designed to look as close to English as possible. Other languages (like C or Assembly) are harder to read but give programmers more direct control over what the computer does. For learning CS concepts — which is what this course is about — Python is ideal.

The AP exam's language

AP pseudocode — a language you'll need to read

The AP CSP exam doesn't test you in Python, JavaScript, or any specific language. Instead, it uses its own simplified notation called AP pseudocode. You'll see it on every question that involves reading or writing code. It's not hard — but it's different enough from Python that it'll trip you up if you encounter it for the first time in April.

The goal of pseudocode is to express programming logic without worrying about any specific language's syntax. Think of it as "the idea of the code" rather than the code itself. The AP exam uses it so that students who learned in Python, JavaScript, Scratch, or any other language all see the same thing.

Here's the same program written in both Python and AP pseudocode, side by side:

Python
# Get a name and greet the user name = input("What is your name? ") age = int(input("How old are you? ")) if age >= 18: print("Hello, " + name) print("You are an adult.") else: print("Hello, " + name) print("You are a minor.")
AP Pseudocode
// Get a name and greet the user name ← INPUT() age ← INPUT() IF age ≥ 18 THEN DISPLAY "Hello, " + name DISPLAY "You are an adult." ELSE DISPLAY "Hello, " + name DISPLAY "You are a minor." ENDIF

The logic is identical. The notation is different. Here's a quick reference for the AP pseudocode constructs you'll see most often on the exam:

AP pseudocode Python equivalent What it does
x ← value x = value Assigns a value to a variable (← is "gets the value of")
DISPLAY x print(x) Prints x to the output
INPUT() input() Gets input from the user
IF condition THEN ... ENDIF if condition: ... Conditional — runs the block only if condition is true
IF ... ELSE ... ENDIF if ... else: ... Two-branch conditional
REPEAT n TIMES ... for i in range(n): ... Repeats a block exactly n times
REPEAT UNTIL condition while not condition: ... Repeats until the condition becomes true
FOR EACH item IN list for item in list: ... Iterates over every item in a list
list[i] list[i-1] Accesses element i — AP lists are 1-indexed, Python is 0-indexed
PROCEDURE name(param) def name(param): Defines a procedure (function)
RETURN value return value Returns a value from a procedure
The one gotcha that trips everyone up

AP pseudocode lists are 1-indexed — the first item is at position 1, not position 0. Python lists are 0-indexed — the first item is at position 0. This difference appears on the exam every single year. When you see list[1] in AP pseudocode, it means the first element. In Python, list[1] means the second element. Don't confuse them.
